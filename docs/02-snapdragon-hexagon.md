# 02. Hexagon NPU в Snapdragon 8 Elite Gen 5

S26 Ultra (международная версия) построен на **Qualcomm Snapdragon 8 Elite Gen 5 for Galaxy** (внутренний номер SM8850). С точки зрения NPU нас интересует именно **Hexagon HTP v81** — тензорный сопроцессор.

## Что обещает Qualcomm

| Параметр              | Snapdragon 8 Elite Gen 5            |
|-----------------------|--------------------------------------|
| HTP architecture      | v81                                  |
| HMX (matrix engine)   | INT8 / INT16 / FP16, ~64×64 MAC/cycle |
| TOPS (INT8 sparse)    | ~100 TOPS заявленные                |
| Частота HTP           | ~1.5 ГГц                             |
| VTCM                  | 16 МБ                                |
| Большая DDR-шина      | LPDDR5X-9600, ~67 ГБ/с               |
| ION-shared buffer     | Да (для нулевого копирования с CPU/GPU) |

«100 TOPS» — это маркетинговое число при INT8 + 2:4 структурной разреженности. Реальная стабильная производительность на плотных INT8-GEMM ближе к 40–55 TOPS, а на FP16 — 15–22 TFLOPS. Этого хватает, чтобы прогнать SD 1.5 целиком за 10–15 секунд (512×512, 20 шагов).

## Архитектура целиком

Snapdragon 8 Elite Gen 5 — это **SoC**, и NPU там не один. Qualcomm называет всё вместе «Qualcomm AI Engine»:

```
                  Qualcomm AI Engine
   ┌──────────────────────────────────────────────┐
   │                                              │
   │  Kryo CPU (Oryon-V3 cores)                   │
   │  - 8 cores: 2P + 6E или 2P + 4P + 2E         │
   │  - SME / SVE2 SIMD, INT8/FP16 на CPU         │
   │                                              │
   │  Adreno 840 GPU                              │
   │  - OpenCL/Vulkan compute                     │
   │  - FP16 GEMM ~7 TFLOPS                       │
   │  - Доступен через QnnGpu backend             │
   │                                              │
   │  Hexagon HTP v81  ◀── это NPU                │
   │  - HMX тензорный сопроцессор                 │
   │  - 16 МБ VTCM                                │
   │  - Доступен через QnnHtp backend             │
   │                                              │
   │  Sensing Hub (low-power AI)                  │
   │  - Always-on модели <10 мВт                  │
   │                                              │
   └──────────────────────────────────────────────┘
```

QNN SDK позволяет адресовать любой из этих исполнителей одним и тем же кодом — меняется только имя shared library (`libQnnHtp.so` / `libQnnGpu.so` / `libQnnCpu.so`). Это удобно для сравнения: в нашем приложении на экране Benchmark по умолчанию запускаются три бэкенда на одной и той же модели.

## Что важно знать про HTP v81

### Quantization stack

HTP считает в INT8 (с накоплением в INT32) и в FP16. Это значит, что любая FP32-модель должна быть проквантована *перед* компиляцией. AI Hub делает это автоматически с режимом `--quantize_full_type w8a16`:
- **w8** — веса в INT8 (per-channel scale)
- **a16** — активации в INT16 (per-tensor scale)

Для генеративных моделей вроде SD это критично — INT8-активации дают видимые артефакты (полосы, цветной шум), поэтому именно `w8a16` стандартный профиль.

### Lockstep с CPU

HTP — это **DSP**, физически отдельный процессор. Когда CPU вызывает `QnnGraph_execute`:
1. Драйвер DSP получает сигнал через FastRPC.
2. На стороне DSP запускается «skel» (skeleton process) — это часть `libQnnHtpV81Skel.so` (мы возим её в APK).
3. Skel читает входы из ION buffer, прогоняет граф, пишет выходы.
4. Сигнал обратно на CPU.

Эта пересылка стоит ~0.5–2 мс fixed cost на каждом вызове. Поэтому батч из 100 крошечных моделей на NPU будет хуже, чем 100 вызовов на CPU. Если модель меньше ~1 ms на NPU — лучше CPU. Если больше ~5 ms — NPU выигрывает уже по wall-time, не только по энергии.

### Power state

HTP в idle потребляет <5 мВт. Под полной нагрузкой — до ~3 Вт. Это означает, что от батареи S26 Ultra (~5000 мА·ч ≈ 19 Вт·ч) при непрерывном инференсе NPU проработает 6–8 часов. Тот же набор инференсов на десктопе с 4080S съест в ~30 раз больше энергии за тот же wall-time.

### Динамические формы

HTP **не любит** динамические формы. Если граф собран под `batch=1, seq=77`, попытка прогнать `seq=123` упадёт с ошибкой. AI Hub компилирует под фиксированные shape. Для SD 1.5 это означает: text encoder = 77 токенов всегда, UNet = 64×64 латент всегда, VAE = 64×64→512×512 всегда.

## Сравнение с RTX 4080 SUPER

|                                | Hexagon HTP v81          | RTX 4080 SUPER              |
|--------------------------------|--------------------------|-----------------------------|
| TOPS INT8 (peak)               | ~100 (sparse) / ~50 dense | ~836 (sparse) / ~418 dense |
| TFLOPS FP16                    | ~20                      | ~104                        |
| Память                         | 16 МБ VTCM + общий DDR  | 16 ГБ GDDR6X                |
| Память bandwidth               | ~67 ГБ/с (DDR-шина SoC) | ~736 ГБ/с                  |
| TDP                            | ~3 Вт                   | ~320 Вт                    |
| TOPS/Вт (INT8 dense)           | ~17                      | ~1.3                        |
| Стоимость                     | в составе телефона       | ~$1000 GPU + ~$500 остальное PC |

Вывод: **по чистой производительности 4080S выигрывает в 8–20× в зависимости от модели**. Но по **TOPS/Вт NPU в ~13× эффективнее**. И поэтому он живёт в телефоне с пассивным охлаждением, а 4080S — в десктопе с тремя вертушками.
